# cortexCic — Project Context

## Language
- **TypeScript only** — no plain JS files. React components use `.tsx`, all other modules use `.ts`.

## Stack
- **Frontend**: Vite + React (TSX), runs locally via Docker
- **Backend**: Firebase Functions (Node.js) — server-side logic, secrets via env vars
- **Database**: Firestore
- **Auth**: Firebase Authentication
- **Storage**: Firebase Storage (file uploads)
- **Hosting**: Firebase Hosting
- **Routing**: React Router
- **State management**: Redux (Redux Toolkit slices)

## Icons & Emojis
- Prefer **Google Material Symbols** (`material-symbols-outlined` icon font)
- No icon libraries (FontAwesome, etc.)

## Styling conventions
- **Bootstrap classes** for layout and utility styles (grid, spacing, buttons, etc.)
- **CSS Modules** (`.module.css`) for component-level styles — each component that needs custom styles creates a `.module.css` file at the same level
- **Global CSS** only for truly generic things:
  - `src/styles/main.css` — imports only, no actual rules
  - `src/styles/partials/` — partial files (e.g. `font.css` for `@font-face` / font imports), all imported in main.css

## Components
- **Btn component** — use for all buttons unless a specific custom design is required. Import from `src/components/Btn/Btn`.

## Firebase environment / secrets
- Secrets and env vars live in Firebase Functions config or `.env` files — never hardcoded
- Frontend Firebase config via `.env` (Vite `VITE_` prefix)
- Backend secrets managed via `firebase-secrets-cli` (`npx cic-index` to add new secrets)
  - Automatically generates `functions/config/secret.ts` when secrets are added
  - Common config vars (REGION, shared settings) in `functions/config/config.ts`

## Layer responsibilities

### `components/`
Global reusable components and hooks only. Anything specific to a single view goes in that view's `cmp/` subfolder instead.

### `api/` folders
- **`src/api/`** (Frontend) — external API integrations for the React app. One file per API when needed.
- **`functions/src/api/`** (Firebase Functions) — server-side API integrations and helpers. One file per API when needed.

### `db/<feature>/`
- **`Feature.ts`** — TypeScript model/interface
- **`featureRepo.ts`** — entry point for Firestore data: `getAll`, `getById`, filtered queries. Used to populate Redux slices.
- **`featureSlice.ts`** — Redux Toolkit slice for this feature

### `views/`
One folder per route. Group folders by model/domain when routes are related. Each view folder contains its `.module.css` and optionally a `cmp/` folder for view-specific sub-components.

### `enums/`
Each file exports:
1. A `const` object with `key === value` (string enum pattern)
2. A `getValue(key)` helper
3. An `isSomeEnum(val)` type-guard

### `serviceWorker.ts`
Only created if push notifications are required.

### Firebase Functions structure (`functions/src/`)
- **`index.ts`** — exports only; routes all functions
- **`config/config.ts`** — REGION constant and shared environment variables
- **`config/secret.ts`** — auto-generated by `firebase-secrets-cli` when secrets are added; never edit manually
- **`api/`** — server-side API integrations (one file per API when needed)
- **`<feature>/`** — feature-specific functions organized by domain

To add a new secret, run `npx cic-index` and the tool will update `secret.ts` automatically.

## Firebase Security Rules

Rules are role-based. Roles (`admin`, `superAdmin`, other) are stored in the user's (permissions) Firestore doc and checked via helper functions. Key access patterns:
- `/users/{uid}` — owner can read/write own doc (except permissions fields); admins can do everything
- `/appConfig` — public read, `superAdmin` write only
- Storage rules follow the same role/auth logic

## Push Notifications (FCM)

Using **Firebase Cloud Messaging**. An FCM token is generated per device and saved in an array field on the user's Firestore doc. Functions use these tokens to send targeted push notifications to the user's devices.

## Toast

Use `react-hot-toast` via the project wrapper — **never import directly from `react-hot-toast`** in feature code.

```typescript
import { toast } from '../../components/toast/toast'; // adjust relative path as needed

// Simple message
toast.success('Saved');
toast.error('Something went wrong');

// With subtitle and/or duration
toast.error('Upload failed', { subtitle: 'File exceeds 10 MB limit', duration: 5000 });
toast.loading('Saving…', { id: 'save-op' });

// Dismiss a specific toast by id
toast.dismiss('save-op');
```

The `ToastOptions` type extends react-hot-toast's native options (`duration`, `position`, `style`, `icon`, `id`, etc.) with an additional `subtitle?: string` field.
When `subtitle` is provided, the toast renders a two-line custom layout. All native options are available in TypeScript autocomplete.
